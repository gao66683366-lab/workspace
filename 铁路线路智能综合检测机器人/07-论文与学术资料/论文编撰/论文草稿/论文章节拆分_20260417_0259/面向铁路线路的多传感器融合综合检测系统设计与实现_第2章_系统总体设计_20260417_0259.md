# 面向铁路线路的多传感器融合综合检测系统设计与实现

## 第2章 系统总体设计

### 2.1 综合检测理念阐述

**（1）综合检测的概念定义**

传统铁路检测装备和方案多以单功能检测为主，即一台设备专职完成一项检测任务。如钢轨探伤仪专职检测钢轨内部伤损，轨距检测仪专职测量轨距值，光学波磨仪专职测量波磨病害。每次检测出动需要使用多套设备，分别上道、分别作业、数据孤立，形成"多套设备、多次出动、多人协同"的低效率模式。

本文提出并定义的"综合检测"是指：利用一套集成化的检测装备，在单次出行中同步完成多种类型的检测任务，所有检测数据在统一的时空基准下生成，具备跨传感器数据融合分析能力的检测模式。

综合检测的核心特征体现在三个维度：

**同源数据：** 同一台设备、同一次出行、同一个里程坐标，同步完成全部检测项目。不同传感器的数据天然具有时间和空间的一致性，无需额外配准即可进行交叉验证。

**一次完成：** 一台设备跑完全部检测流程，作业组织复杂度从"多人多天"降为"单人一次"，大幅降低检测的组织和协调成本。

**融合判定：** 多种传感器的异构数据在统一框架下融合分析，单一维度的检测结果可通过其他维度交叉验证，判级准确性显著优于单一传感器独立判定。

**（2）综合检测 vs 单一功能检测的本质区别**

综合检测与单一功能检测的本质区别，不在于传感器数量的多少，而在于系统设计理念的根本差异。单一功能检测的思维是"为每种检测任务配备一套专用设备"，综合检测的思维是"构建一个统一的数据采集和融合分析平台，共享传感器和计算资源"。

从系统层面看，综合检测实现了三个层面的深度融合：

**硬件层融合：** 多种传感器共享同一运动平台和电源系统，传感器之间通过统一的时间基准和空间坐标实现有机协同，而非简单拼凑。

**数据层融合：** 所有传感器的原始数据在统一时空基准下存储和管理，通过帧编号、里程坐标、时间戳三码对齐，实现跨传感器的数据关联和融合分析。

**算法层融合：** 不同传感器数据按预设权重进行加权投票融合，每种病害类型均有其对应的最优权重配置，融合结果优于任何单一传感器独立判定的结果。

**（3）本系统的综合检测能力概览**

本文设计的综合检测系统，以一台检测机器人为载体，单次出行可同步完成8项检测功能，覆盖轨面缺陷、道钉螺栓状态、钢轨廓形、波磨病害、焊缝缺陷、轨距值、水平偏差、高低不平顺等主要检测项目。8项检测功能共享同一运动平台、同一计算系统和同一数据存储系统，数据格式统一、时空基准统一，具备天然的数据融合分析条件。

### 2.2 系统功能需求分析

**（1）检测功能需求**

根据铁路线路检测的相关标准和实际运维需求，本系统需满足以下检测功能需求：

| 序号 | 检测功能 | 检测内容 | 输出指标 |
|---|---|---|---|
| 1 | 轨距检测 | 左右钢轨内侧距离 | 轨距值(mm)、偏差值(mm)、超限标记 |
| 2 | 水平检测 | 左右轨高差 | 水平偏差值(mm)、左右倾斜角(°)、平顺性指标 |
| 3 | 高低检测 | 纵向不平顺 | 高低不平顺值(mm)、长波偏差、短波偏差 |
| 4 | 轨面缺陷检测 | 裂纹、掉块、凹陷等 | 缺陷位置(里程坐标)、缺陷类型、损伤等级(1~3级) |
| 5 | 道钉/螺栓检测 | 缺失、松动、歪斜 | 数量统计、里程分布 |
| 6 | 3D钢轨轮廓检测 | 轨头廓形、磨耗量 | 磨耗量(mm)、横移量(mm)、廓形偏差(mm)、超限标记 |
| 7 | 钢轨波磨检测 | 波磨病害 | 波长(mm)、波深(mm)、分布里程、劣化程度 |
| 8 | 钢轨焊缝检测 | 焊缝缺陷 | 缺陷类型、等级、里程位置 |

| 性能指标 | 需求值 | 说明 |

| AI推理时延 | < 1 s | 实时输出检测结果 |

| 轨距测量精度 | ±1 mm | 满足规范要求 |
| 水平/高低精度 | ±0.1° | 满足轨道几何参数测量需求 |
| 连续作业时间 | ≥ 2 h | 单次充电可支持完整检测任务 |

**（3）数据管理需求**

| 需求项 | 说明 |
|---|---|
| 本地存储 | 检测数据实时写入本地SSD，断网不丢失 |
| 云端同步 | 移动网络实时回传，云端双重备份 |
| 数据格式 | 统一Frame ID+里程+时间戳，全流程可追溯 |
| 数据检索 | 支持按里程区间、时间段、病害类型等多维检索 |
| 报告生成 | 任务结束后自动生成标准化检测报告 |

**（4）人机交互需求**

| 需求项 | 说明 |
|---|---|---|
| 操作简便 | 非专业人员经简单培训即可操作 |
| 启机流程 | 3步以内完成出发准备 |

| 异常告警 | 本地声光+云端推送，多级分级响应 |
| 操控冗余 | 触摸屏、按钮、遥控器多种操控方式互为备份 |

### 2.3 系统总体架构

**（1）系统架构总览**

本文设计的综合检测系统由硬件系统和软件系统两大部分组成，二者协同实现全自动、实时、综合的铁路线路检测功能。

硬件系统采用五层架构设计，自下而上依次为：供电层提供整机动力来源，运动层实现机器人行走功能，传感层负责多源数据采集，计算层运行AI推理和融合算法，交互层提供人机操作界面。

软件系统采用三层架构设计，自内而外依次为：边缘计算层部署于工控机上，负责实时数据采集、AI推理和本地存储；通信层通过三网物理隔离实现控制指令下发、采集数据回传和远程监控；云平台层部署于远程服务器上，负责数据存储、大数据分析、健康评估和养护决策。

**（4）三网物理隔离设计**

控制指令、采集数据、远程传输三路通信在物理层面完全隔离，各行其道、互不干扰：

| 网络类型 | 物理介质 | 协议 | 带宽 | 实时性 | 用途 |
|---|---|---|---|---|---|
| 控制网（工业总线） | 双路CAN FD | CANopen over EtherCAT | 100 Mbps | 硬实时 ≤1ms | 伺服驱动/电机控制/急停 |
| 采集网（千兆以太网） | 千兆工业以太网 | GigE Vision + TCP/IP | 1000 Mbps | 软实时 ≤10ms | 6路相机+2路激光实时采集 |
| 传输网（移动网络） | 4G/5G无线 | MQTT + TLS | 50~500 Mbps | 非实时秒级 | 数据上传/远程监控/模型推送 |

三块独立网卡、三个独立MAC地址、三套独立PHY芯片，任一网络故障不影响另外两张网的正常运行。

### 2.5 系统工作流程

**（1）任务前准备流程**

```
设备到场 → 设备就位（4轮自动对准轨道）→ 开机自检
   ↓
传感器自检（相机/激光/陀螺仪/测距仪）→ 触摸屏状态确认
   ↓
任务参数配置（里程区段/检测速度）→ 一键出发
```

**（2）检测任务执行流程**

```
机器人沿轨道定速巡航（1m/s）→ 多传感器同步采集
   ↓
数据实时融合 → AI推理秒级输出 → 缺陷即检出即报警
   ↓
本地SSD实时存储 → 4G/5G实时回传云端
   ↓
里程到达目标终点 → 自动减速停车 或 一键定点返航
   ↓
数据完整上传 → 任务闭合
```

**（3）多维度数据融合判级流程**

```
图像帧（2D相机） ──┐
                  ├──→ 时钟同步 ──→ 时空对齐
点云帧（3D激光） ──┤
                  │    ↓
姿态帧（陀螺仪） ──┘    ↓
                     空间关联 ──→ 融合节点
                                          ↓
                              加权投票 ──→ 综合判级
                                          ↓
                              输出：缺陷类型 + 里程坐标
                                    + 等级 + 置信度
```

在需求分析方面，明确了系统需满足的8项检测功能需求、实时性、精度、存储、人机交互等性能和功能需求。在架构设计方面，给出了硬件系统五层架构（供电层→运动层→传感层→计算层→交互层）和软件系统三层架构（边缘计算层→通信层→云平台层）的详细设计方案，创新性地提出了控制网、采集网、传输网三网物理隔离的通信架构。在技术参数方面，给出了运动平台、感知系统、计算系统、供电系统、通信软件等各子系统的详细技术指标和系统综合性能指标。本章为后续章节的详细设计奠定了总体框架基础。
